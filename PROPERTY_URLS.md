# Per-Property URLs Guide

PropIntel uses a full-screen slide-over (not a separate URL) for property
detail pages. This guide explains how that works, how the routing is structured,
and how to evolve it toward dedicated `/property/[id]` URLs if needed.

---

## How Property Views Work Today

Property detail pages are **not** separate routes. They are a full-screen
overlay rendered inside the main dashboard.

When a user clicks a property card:

1. `Dashboard.tsx` calls `analyze(property)` (a `useCallback`).
2. `analyze` fires a `POST /api/analyze-property` request with `{ propertyId }`.
3. While the request is in flight, a skeleton loader shows in the panel.
4. The response (`AnalysisResult`) is stored in `analysis` state.
5. `AnalysisPanel` renders as a full-screen overlay with the photo carousel,
   all metrics, and the action plan.

The URL does not change. The back button (`close()`) hides the panel with a
CSS transition, then clears the state after 300 ms.

---

## The Analysis API Route

```
POST /api/analyze-property
Body: { "propertyId": "prop_001" }
Returns: AnalysisResult
```

Property IDs are strings like `"prop_001"` through `"prop_050"`. They are
defined in `data/properties.json` and accessed via `lib/data.ts`.

```ts
// lib/data.ts
export function getPropertyById(id: string): Property | undefined {
  return properties.find((p) => p.id === id);
}
```

The API route validates the `propertyId`, looks up the property, fetches the
live FRED mortgage rate, runs the analysis, and returns JSON. See
`app/api/analyze-property/route.ts`.

## How to Add a Dedicated `/property/[id]` Route

If you want shareable URLs (e.g. `https://propintel.com/property/prop_001`),
here is the minimal change set. This is a Next.js App Router dynamic segment.

### 1. Create the route directory and page file

```
app/
  property/
    [id]/
      page.tsx
```

### 2. `app/property/[id]/page.tsx`

```tsx
import { notFound } from "next/navigation";
import { getPropertyById } from "@/lib/data";
import { analyzeProperty } from "@/lib/analysisService";
import { getMarket } from "@/lib/market";
import { LanguageProvider } from "@/components/LanguageProvider";
import { FavoritesProvider } from "@/components/FavoritesProvider";
import AnalysisPanel from "@/components/AnalysisPanel";

// Next.js 16 passes params as a Promise — await it.
export default async function PropertyPage({
  params,
}: {
  params: Promise<{ id: string }>;
}) {
  const { id } = await params;
  const property = getPropertyById(id);
  if (!property) notFound();

  const market = await getMarket();
  const analysis = analyzeProperty(property, market.mortgage30.value);

  return (
    <LanguageProvider>
      <FavoritesProvider>
        <AnalysisPanel property={property} analysis={analysis} onClose={null} />
      </FavoritesProvider>
    </LanguageProvider>
  );
}
```

Note: `AnalysisPanel` is a client component that currently expects `onClose`
to be a function. For the standalone page, pass `null` and add a link back to
`/` instead of a close button.

### 3. Generate static paths (optional, for faster initial load)

```ts
// In app/property/[id]/page.tsx
import { getAllProperties } from "@/lib/data";

export function generateStaticParams() {
  return getAllProperties().map((p) => ({ id: p.id }));
}
```

This pre-renders all 50 property pages at build time. Without this, pages are
rendered on-demand (still fast, just not pre-built).

### 4. Add metadata for SEO

```ts
export async function generateMetadata({
  params,
}: {
  params: Promise<{ id: string }>;
}) {
  const { id } = await params;
  const property = getPropertyById(id);
  if (!property) return {};

  return {
    title: property.address,
    description: `Investment analysis for ${property.address}. ${property.beds}bd/${property.baths}ba, $${property.price.toLocaleString()}.`,
    openGraph: {
      images: [property.imageUrl],
    },
  };
}
```

## URL Structure

| URL | What it shows |
|---|---|
| `/` | Dashboard (all 50 listings, search, filters, map) |
| `/property/prop_001` | Standalone page for listing 001 |
| `/api/analyze-property` | POST-only API, returns JSON |
| `/sitemap.xml` | Generated by `app/sitemap.ts` |
| `/robots.txt` | Static file in `public/` |

## How New Properties Are Indexed

Adding a new property to the dataset requires two steps:

1. Add an entry to `data/properties.json` (or re-run `scripts/build-properties.mjs`).
2. Redeploy. The new property is immediately available via:
   - The dashboard (auto-included in `getAllProperties()`).
   - The API route (auto-found by `getPropertyById()`).
   - A `/property/[id]` route if you add one (auto-generated if using `generateStaticParams`).

No router configuration, no hard-coded lists.

## SEO Considerations

### Current state

- The dashboard (`/`) has full OG metadata, Twitter cards, and JSON-LD schema.
- Individual properties do not have dedicated URLs, so they are not indexable.

### When you add `/property/[id]` routes

- Each page gets its own `<title>` and description via `generateMetadata`.
- Use the primary property photo as the OG image (already in `property.imageUrl`).
- Add each property URL to the sitemap (update `app/sitemap.ts`):

```ts
import { getAllProperties } from "@/lib/data";

export default function sitemap() {
  const properties = getAllProperties();
  return [
    { url: BASE_URL, changeFrequency: "daily", priority: 1 },
    ...properties.map((p) => ({
      url: `${BASE_URL}/property/${p.id}`,
      changeFrequency: "weekly" as const,
      priority: 0.8,
    })),
  ];
}
```

- Consider adding Property/RealEstateListing JSON-LD structured data for each
  page to help Google surface listing details in search results.

## Testing Individual Property Views

### Current overlay (no URL)

1. Open the dashboard.
2. Click any property card.
3. The full-screen panel opens.

### Future `/property/[id]` route

```bash
# Start dev server
npm run dev

# Open in browser
open http://localhost:3000/property/prop_001

# Test a missing ID (should show 404)
open http://localhost:3000/property/prop_999
```

### Testing the API directly

```bash
# Any property ID from data/properties.json
curl -X POST http://localhost:3000/api/analyze-property \
  -H "Content-Type: application/json" \
  -d '{"propertyId": "prop_001"}'

# Missing ID (should return 404)
curl -X POST http://localhost:3000/api/analyze-property \
  -H "Content-Type: application/json" \
  -d '{"propertyId": "prop_999"}'

# Malformed body (should return 400)
curl -X POST http://localhost:3000/api/analyze-property \
  -H "Content-Type: application/json" \
  -d '{}'
```

## How to Customize Property Pages

All property display logic lives in `components/AnalysisPanel.tsx`. It receives
a `Property` and an `AnalysisResult` and renders everything: the photo
carousel, fact grid, score bars, highlights, risks, action plan, and the
contact CTA.

To change the layout or add new sections, edit `AnalysisPanel.tsx`. The data
contracts (`Property` and `AnalysisResult` in `lib/types.ts`) define what
fields are available.
